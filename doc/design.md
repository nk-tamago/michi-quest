# MichiQuest アプリケーション設計書

## 1. システム概要
MichiQuestは生成AI（Gemini API）を活用したツーリング・ミッション生成＆報告アプリケーションです。
ユーザーはAIから提示されたミッション（場所や対象物）に基づき探索を行い、写真（GPS情報付き）で報告することでAIから評価を受けます。

## 2. アーキテクチャ構成
*   **Frontend Framework**: React (Vite)
*   **Styling**: Tailwind CSS
*   **状態管理**: React Hooks (`useState`, `useEffect`), カスタムフック (`useLocalStorage`) によるブラウザ内永続化
*   **地図API**: `react-leaflet`, `leaflet` (OpenStreetMap)
*   **画像/Exif処理**: `exifr` (GPS座標抽出), カスタムCanvasリサイズ処理
*   **PWA対応**: `vite-plugin-pwa`

## 3. 主要コンポーネント設計
*   **App.jsx**:
    *   全画面の親コンポーネント。タブ構成（チャット、地図、設定）の切り替え。
    *   `chatSessions`（会話履歴・ミッション状態のリスト）や現在地のグローバルな状態管理。
*   **ChatThread.jsx**:
    *   AIとのメインのやり取りを行うスレッド型チャットUI。
    *   画像アップロード機能、Exif解析トリガー、Gemini APIへのリクエスト処理を包含。
*   **MapInteractive.jsx**:
    *   Leafletベースの地図インタラクション。
    *   「ミッションエリア（円）」「現在地（青マーカー）」「写真のアップロード地点（画像サムネイル付きマーカー）」を統合表示。
*   **Settings.jsx**:
    *   システムプロンプトのカスタマイズ、APIキー、AIキャラクターの感情別アバター設定。
*   **Sidebar.jsx**:
    *   過去のツーリング（セッション）の履歴一覧と切り替え。

## 4. データ構造 (LocalStorage)
*   `chatSessions`: セッション情報の配列。
    *   型定義イメージ: `Array<{ id: number, title: string, history: Array<Message>, currentMission: string, currentMissionArea: { lat, lng, r } }>`
*   `geminiApiKey` / `geminiAiModel`: 認証・モデル設定
*   `aiPrompt1` / `aiPrompt2`: ミッション生成・報告受付用システムプロンプト
*   `aiAvatarData` 他: キャラクター画像のBase64データ

## 5. フェーズ3: AI高度化・ゲーム要素の設計プラン

フェーズ3では、アプリの「遊び」としての要素を強化し、ミッションの精度を向上させます。

### 5-1. ハルシネーション（存在しない目的地）対策機能
現在、AIが自由に地名や目的地を生成するため、実在しない場所が指定される課題があります。これに対して2つのアプローチを用意・実装します。

1.  **目的地リスト管理（確実な制御）**:
    *   設定画面に「目的地候補リスト（例：道の駅、特定の神社リストなど）」を入力できるテキストエリアを追加。
    *   ミッション生成のリクエスト時、ユーザーがいずれからのリストを選択した場合、裏側で結合し「以下のリストの場所から1つ選びなさい」とAIを制御します。
2.  **外部APIによる実在チェック検証（柔軟な探索）**:
    *   特定のリストを使わずAIに自由に指定させる場合、AIが出力した目的地名（`[AREA: {"lat": ..., "lng": ..., "name": "..."}]` 等のフォーマットに拡張）を抽出し、**OpenStreetMap の Nominatim API（無料のジオコーディングAPI）** にバックグラウンドでリクエストを送信します。
    *   Nominatim API から座標や施設情報が返ってくれば「実在する」と判定してそのまま表示し、見つからない・想定外の場所（海外など）になった場合はエラーハンドリングを行い、AIに「再生成」を要求するオートリトライ機構を設けます。
    *   APIの利用制限（1秒間に1リクエスト等）に留意した設計とします。

### 5-2. 動的生成「称号コレクション」機能
当初予定していた累計スコアに応じた固定ランク制を改め、**AIによる動的な称号生成**を導入します。これにより上限なく多様な称号を集める楽しみ（コレクション要素）を提供します。
*   **称号のAI生成**: 完了報告時のプロンプトで、AIに `[SCORE: 100]` とともに、ミッション内容や評価に応じたユニーク（かつ少しクスッとくるような）な称号を `[TITLE: 〇〇の覇者]` のような形式で出力させます。
*   **称号・スコアの保存**: パースした称号文字列と獲得スコアを、セッションデータ（`chatSessions`）の一部として、またはユーザーの「獲得称号リスト（`titleCollection`）」として LocalStorage に保存します。
*   **称号コレクション画面**: サイドバー（または別途「称号」タブ等）に、これまでにAIから授与された独特な称号のリストを表示し、眺めて楽しめる仕組みを用意します。
