# 04. 機能設計 (Functional Design)

## 1. チャット対話・ミッション生成フロー
1.  **セッション初期化と挨拶**: サイドバー等から新規セッションを作成した際、チャット履歴の初期状態として、定義されたオペレータ（ミチ・ノマ）のランダムな定型文挨拶が1件挿入された状態で開始する。
2.  **自動ミッション生成**: 初期挨拶の表示後、ユーザーの入力を待たずにシステムが自動でGemini APIを呼び出し、ミッションを生成してユーザーに提示する。
    1.  **プロンプト構築 (`api.js`)**:
        *   `App.jsx` から渡された `basePrompt` (基本性格), `prompt1` (ミッション生成用), `prompt2` (画像判定用) などをコンテキストとして組み立て。
        *   システムプロンプト内に `[AREA: {"lat":35.xxx, "lng":139.xxx, "r":100, "name":"地名"}]` 形式での出力を要求。
    2.  **API通信**: Gemini APIに対してリクエストを実行。
    3.  **レスポンスパース (`App.jsx` / `ChatThread.jsx`)**:
        *   応答から `[AREA: ...]` を正規表現で取り出し、地図コンポーネント用の `currentMissionArea` ステートに保存。該当文字列はチャット画面では置換（非表示）とする。
        *   同様に `[TITLE: ...]` (称号), `[SCORE: ...]` (得点) をパースし、コレクションとしてLocalStorageに保存。
        *   **クリア判定と演出**: **60点以上**のスコアを獲得した際に、当該セッションの状態（`isCleared`）を true に更新し永続化する。更新と同時にチャット画面上に「MISSION CLEARED!」等のアニメーションをオーバーレイ表示し、サイドバーの一覧にも「CLEAR!」バッジを表示する。

## 2. ハルシネーション（目的地架空生成）防止フロー
*   **概要**: ミッション生成時に `[AREA]` タグが含まれていた場合、事前にOpenStreetMapなどで実在判定を行う（フェーズ3設計）。
*   **ロジック**:
    1.  AIが「富士山頂上のカフェ」などを生成し、`[AREA: {"name": "富士山頂上のカフェ"}]` を出力。
    2.  フロントエンド側で抽出された `name` を使って Nominatim API を叩く。
    3.  結果が空 (API return `[]`) である場合、フロントエンドはエラーを検知し、裏側で「その場所は存在しません、別の実在する場所でミッションを作り直してください」とGeminiに再リクエストを投げる (Auto Retry機構)。
    4.  検証を合格した場所のみがユーザーのチャット画面に表示される。

## 3. リプレイモード（フェーズ4）
*   **UI構成**:
    *   サイドバーのセッション・アイテムから「リプレイ開始」を実行。
    *   `App.jsx` で `isReplayMode = true` となり、`ChatThread.jsx` にPropとして渡る。
    *   チャット下部の「送信入力欄」の領域が、動的に「リプレイ・コントローラー（▶️やシークバーのようなUI）」に置換される。
*   **制御ロジック**:
    *   `ChatThread.jsx` 内に `replayIndex` ステートを持つ。
    *   表示する履歴を `chatHistory.slice(0, replayIndex + 1)` として描画。
    *   右矢印キーによるインクリメント、オートプレイ（`setInterval`）による動的インクリメントに対応。
    *   **クリア演出の再現**: `isCleared` が true のセッションをリプレイした場合、最終インデックスまで再生が到達した時点で、手動プレイ時と同様のクリアアニメーションを再表示する。

## 4. 画面レイアウト・ルーティング
*   React Router等のライブラリは使用せず、`App.jsx` の `currentTab` Stateによってコンポーネント (`ChatThread`, `MapInteractive`, `Settings`) を切り替えるSPA（Single Page Application）として構築。
*   スマホ利用（バイク搭載のスマホマウント状態など）を想定し、画面の大部分をチャットまたは地図UIに割き、ヘッダー・タブナビゲーションのみを常に表示する。
*   **チャットUI仕様**:
    *   AIアバター画像は、表情視認性向上のため基本サイズを `w-28 h-28 (112px)`、MDサイズ（PC）で `w-32 h-32 (128px)` と大きく設計する。
    *   APIタイムアウト時や通信に失敗した場合、必ずローディング表示を取り下げて適切なエラーテキストを表示し、フリーズを防止する。
