# 04. 機能設計 (Functional Design)

## 1. チャット対話・調査生成フロー
1.  **セッション初期化と挨拶**: サイドバー等から新規セッションを作成した際、チャット履歴の初期状態としては「空（0件）」で開始する。チャット画面コンポーネントが初めてマウントされた際に一度だけ、オペレータ（ミチ・ノマ）のランダムな定型文挨拶を生成し、システム共通の**「時間差メッセージ追加キュー (`addMessagesWithDelay`)」**に乗せて順次画面に表示する（React Strict Mode下での重複生成防止のため1度限りの実行フラグを管理する）。
2.  **自動調査作成**: 初期挨拶がキューに入り表示された後、ユーザーの入力を待たずにシステムが自動でGemini APIを呼び出し、最初の調査対象を生成してユーザーに提示（同じディレイキュー経由で表示）する。
    1.  **プロンプト構築 (`api.js`)**:
        *   `App.jsx` から渡された `basePrompt` (基本性格), `prompt1` (調査生成用), `prompt2` (画像判定用) などをコンテキストとして組み立て。
        *   システムプロンプト内に `[AREA: {"lat":35.xxx, "lng":139.xxx, "r":100, "name":"地名"}]` 形式での出力を要求。
    2.  **API通信**: Gemini APIに対してリクエストを実行。
    3.  **レスポンスパース (`App.jsx` / `ChatThread.jsx`)**:
        *   応答から `[AREA: ...]` を正規表現で取り出し、地図コンポーネント用の `currentMissionArea` ステートに保存。該当文字列はチャット画面では置換（非表示）とする。また、調査記録のタイトルをこの目的地の名称（例: `📍東京タワー`）に更新する。
        *   同様に、写真判定時は `[GRADE: ...]` (評価グレード: 1〜3) と `[INSIGHT: ...]` (知見のJSON形式データ) をパースし、知見を `michi-field-notes` に保存・蓄積する。
        *   また、取得した評価グレードや、ユーザーの自発写真送信等の行動履歴に応じてミチ・ノマからの「信頼度」を算出し、`michi-trust-score`（通算スコアの代替）として増減させる。
        *   **リザルトカード表示（★グレード）**: 判定結果はシステムメッセージとしてチャット内にリザルトカード形式で挿入する。カードはダークレザー調（`#4E342E`〜`#3E2723`グラデーション）で、ゴールドのアクセントライン、「ミチ・ノマの評価」ヘッダー、Lucide `Star`アイコン（48px）×3の点灯式で★を表現し、知見獲得時はタイトルを『』で囲んで表示する。
        *   **自発写真の知見獲得バッジ**: ミッション外の自発写真から知見が得られた場合も、「Spontaneous discovery!」と表示するバッジ型のシステムメッセージをチャットに挿入する。
        *   **リザルトの段階的表示フロー**: AIが画像を判定後、一気に結果を出力するのではなく、①AIのタメ言葉（PRELUDE）を先行表示 → ②「評価を受け取る」等ユーザーのアクションボタン表示 → ③ボタン押下でディレイを挟みつつ「★グレード評価カード」および「AIの追加労いコメント」を順次フェードインさせる。これにより会話の情緒とロールプレイの没入感を高める。
        *   **クリア判定とスタンプ演出**: **★★☆（GRADE 2/60点相当）以上**の評価を獲得した際に、当該セッションの状態（`isCleared`）を true に更新し永続化する。このクリア達成の演出として、全画面オーバーレイのポップアップ等は使用せず、段階的表示の最後に出現する「AIの追加労いコメント」のチャットバブル内に、「ミッションクリア（よくできました 等）」を示すCSS/SVGアニメーション付きのスタンプ風画像を配置する。サイドバーの一覧では吹き出しアイコン（`MessageSquare`）の右下にチェックマーク（`CheckCircle`）を重ねて表示する。また、このタイミングのAIの発言オブジェクト自身に `isClearMessage: true` と `clearedTitle` などの情報を記録し、リプレイ時に正確なタイミングでクリア演出を再現できるようにする。
        *   **インラインエモーションによる吹き出し分割**: AIの応答テキスト内に複数の `[Emotion: xxx]` タグが含まれている場合、テキストを感情ごとに分割し、それぞれ独立したチャットメッセージ（吹き出し）として連続で履歴に追加・表示する。これにより、一つの応答内で表情が変化しながらテンポよく語りかける演出を実現する。
    4.  **ナビゲート対応（通常会話・自発写真・調査再生成）**: 調査進行中の助手の発言に対しては、Gemini APIを使用して探究者としての応答（雑談、ヒント提示など）を生成しチャットに表示する。
        *   **自発写真への対応**: ユーザーが調査目的外の写真を任意に送信した場合、AIはそれを独自に学術分析し、有益であれば `[INSIGHT]` タグ付きで応答を返しフィールドノートに記録させる。この際グレードの判定は行わないが、写真の有用性に応じて信頼度が上下する。
        *   **ミッション変更対応**: ユーザーから調査対象変更の要求等があり、システムの判断で新しい場所に切り替える場合、AIは最初渋りつつも最終的に要求を受け入れ、応答に `[AREA: {"lat":..., "lng":..., "name":"...", "r":...}]` を含める。
        *   フロントエンド (`ChatThread.jsx`) は応答内に `[AREA]` タグを検知した場合、Nominatim API での実在判定を通した上で、地図の目的地 (`currentMissionArea`) と `currentMission` ステートを新しいものへ更新する。AREAタグを含まない場合は状態を更新せず会話のみ進行する。

## 2. ハルシネーション（目的地架空生成）防止フロー
*   **概要**: 調査生成時に `[AREA]` タグが含まれていた場合、事前にOpenStreetMapなどで実在判定を行う（フェーズ3設計）。
*   **フェーズ3導入前のフォールバック制御**: AIが架空の目的地（緯度経度が提供されないデータなど）を出力した場合や、アップロードした写真のExif情報が不正だった場合、地図コンポーネントにおける描画エラー（クラッシュ）を防ぐため以下の対策を行う。
    *   `lat` / `lng` が存在しない（数値でない）場合は対象エリアの円形（Circle）マーカー等の地図上表示をスキップする。
    *   地図全体を移動させる中心座標（`center`）に対しても、渡された座標配列が不正な場合（`[NaN, NaN]`など）は強制的に初期位置（東京駅周辺など）へフォールバックさせ、アプリの異常終了を回避する。
    *   現在地や写真によるピン（`Marker`）を描画する際にも、座標配列が有効な数値かを検証し、無効な場合はマーカー単体の描画をスキップして地図全体のクラッシュを防ぐ。

## 7. UI演出仕様（フェーズ6 Phase 2/3）

### 7.1 信頼度ニコちゃんマーク変動アニメーション
*   **対象**: `Sidebar.jsx` 内の信頼度エリア
*   **トリガー**: `trustScore` の値が変化した際に `useEffect` で検知
*   **UP時**: `pop-bounce` アニメーション（scale拡大 → バウンス → 元サイズ、0.6秒）
*   **DOWN時**: `pop-shake` アニメーション（左右回転 + グレースケールフィルタ、0.5秒）
*   **アクセシビリティ**: `prefers-reduced-motion: reduce` 時はすべてのアニメーションを無効化

### 7.2 リザルトカード（★グレード判定表示）
*   **対象**: `ChatThread.jsx` 内の `msg.type === 'evaluation'` のシステムメッセージ
*   **デザイン**: ダークレザー調カード、ゴールドのグラデーションライン、セリフフォント
*   **構成要素**:
    -   「ミチ・ノマの評価」ヘッダーテキスト
    -   Lucide `Star` アイコン（48px）×3。 獲得した★の数に応じてゴールド色 + ドロップシャドウで点灯、未獲得は暗い色で表示
    -   知見タイトル（ある場合）をセリフフォント `text-xl` で『』囲み表示
*   **アニメーション**: `animate-fade-in-up` でフェードイン

### 7.3 自発発見バッジ
*   **対象**: `ChatThread.jsx` 内の `msg.type === 'insight'` のシステムメッセージ
*   **デザイン**: ダークレザー帯 + 左端にゴールドのアクセントバー
*   **構成要素**: Award アイコン + 「Spontaneous discovery!」ラベル + 知見タイトル

### 7.4 フィールドノートモーダル（知見コレクション画面）
*   **対象**: `Sidebar.jsx` 内のモーダル
*   **デザインテーマ**: アナログ革張り手帳風
*   **構成**:
    -   **外装（カバー）**: ダークブラウン(`#4E342E`) + ステッチ風ダッシュボーダー
    -   **中面（ページ）**: クラフト紙調 (`#FDF5E6`) + 罫線背景（`repeating-linear-gradient`）
    -   **各知見カード**: 白背景 + セリフフォント、★/☆バッジ（`GRADE:` テキストなし、星マークのみ）、タイトル・詳細・場所・日付
    -   **空状態**: 「まだ知見が記録されていません…」をイタリック体で表示
*   **ロジック**:
    1.  AIが「富士山頂上のカフェ」などを生成し、`[AREA: {"name": "富士山頂上のカフェ"}]` を出力。
    2.  フロントエンド側で抽出された `name` を使って Nominatim API を叩く。
    3.  結果が空 (API return `[]`) である場合、フロントエンドはエラーを検知し、裏側で「その場所は存在しません、別の実在する場所で調査対象を作り直してください」とGeminiに再リクエストを投げる (Auto Retry機構)。
    4.  検証を合格した場所のみがユーザーのチャット画面に表示される。

## 3. 画像報告・アップロード機能
*   **調査時の入力UI仕様**: ユーザーが写真報告用のカメラボタンを押した際、「カメラで撮影する」と「写真ライブラリから選ぶ」という2つの選択肢（メニュー）をフロントエンド上で提示する。
    *   **カメラ撮影**: `<input type="file" accept="image/*" capture="environment">` を動的に生成して起動し、カメラアプリを直接立ち上げる。
    *   **写真選択**: `<input type="file" accept="image/*">` (`capture` 属性なし) を動的に生成して起動し、OS標準のファイル/ライブラリ選択ダイアログを表示する。
*   **画像処理フロー**: 選択されたファイルからExif情報（GPS座標）を抽出し、画像判定に利用する。APIへ送信する前には長辺1024px等へクライアントサイドでリサイズを行う。
*   **設定画面（アバター）の手動クロップUI**:
    *   `Settings.jsx` にて画像を選択時、または「画像を調整」ボタン押下時、全画面モーダルコンポーネント（`ImageCropper.jsx` / `react-easy-crop`使用）を表示する。
    *   ユーザーによるドラッグ・ズーム操作で円形クロップの枠を合わせ、Canvas API を用いて指定領域のみを正方形（256x256 等）で切り出し、Base64 で保存する。既存のデフォルト画像や設定済み画像に対しても再調整が可能。

## 4. リプレイモード（フェーズ4）
*   **UI構成**:
    *   サイドバーのセッション・アイテムから「手動リプレイ」または「自動リプレイ」を選択して開始。
    *   `App.jsx` で `isReplayMode = true` および `isAutoReplayMode = true/false` となり、`ChatThread.jsx` にPropとして渡る。
    *   プレイ動画収録（スクリーン録画）のリアルさを追求するため、リプレイ専用のコントロールパネルは表示しない。**通常時と同じテキスト入力エリアやカメラボタンを偽装表示**する。
    *   リプレイモード終了用の目立たないボタン（画面右上など）を配置する。
*   **制御ロジック**:
    *   `ChatThread.jsx` 内に `replayIndex`, `isReplayTyping` (AI入力中), `replayInputText` (ユーザー自動入力文字列) ステートを持つ。
    *   表示する履歴を `chatHistory.slice(0, replayIndex)` (現在の実履歴) として描画し、入力中のものは未送信として扱う。
    *   **ユーザー発言の入力演出**: 次に表示されるメッセージがユーザーテキストの場合、テキストエリア（`textarea`）に元のメッセージを一括表示する。画像が含まれる場合は画像プレビューエリアに画像を表示する。その後、わずかなディレイ（約1秒）の後に送信されたように画面へ反映する（`replayIndex`インクリメント）。
    *   **AI発言の遅延演出（入力中アニメーション）**: 次に表示されるメッセージがAIのものである場合、即座に表示せず `isReplayTyping = true` にして「入力中...」インジケータを一定時間（1.5秒程度）表示した後、メッセージを表示（`replayIndex`をインクリメント）する。
    *   **隠し操作（送信ボタンの挙動）**: コントローラーの代わりに疑似表示されている「送信ボタン」は、現在のモードによって挙動が変わる。
        *   **手動再生モード**: 送信ボタンを押すたびに、「次」のメッセージの再生（ユーザー入力表示、またはAIの入力中アニメーション開始）を実行する。
        *   **自動再生モード**: 再生中に送信ボタンを押すと再生一時停止（Pause）となり、停止中に押すと再び自動再生（Play）が開始されるトグルとして働く。
    *   **連続再生時のテンポ最適化**: AIの「入力中...」アニメーションは、直前のメッセージがユーザー発言であった場合や一定の区切りでのみ表示し、AIのメッセージが連続する場合（感情タグにより分割された吹き出しが連続して表示されるケース等）は、待機演出をスキップして即座に次の吹き出しを描画する。
    *   **クリア演出とインラインエモーションの再現**: 
        *   チャット履歴内に保持されている `isClearMessage: true` を持つメッセージに到達した瞬間に、スタンプ画像アニメーションを再表示する。
        *   分割された各メッセージごとに設定された感情情報をもとにアバターの表情を切り替える構成とし、リプレイでも生きた対話を完全に再現する。

## 5. データエクスポート・インポート機能
*   **概要**: ユーザーがローカルストレージに蓄積した調査記録、設定データ、獲得称号などをJSON形式のファイルとしてエクスポートし、他端末またはブラウザの再インストール後にインポートして復元できるようにする。
*   **エクスポート仕様**:
    *   `Settings.jsx` 等の設定画面に「データのエクスポート」ボタンを配置する。
    *   ボタン押下時に、LocalStorage上の対象となるキー（`michi-db-history`, `michi-custom-character`, `michi-field-notes`, `michi-trust-score` など）の値をひとまとめのJSONオブジェクトとして抽出する。
    *   抽出したデータを `Blob` オブジェクトに変換し、ブラウザのダウンロード機能を利用して `michiquest-backup-YYYYMMDD_HHMMSS.json` のようなファイル名でローカルに保存させる。
*   **インポート仕様**:
    *   設定画面に「データのインポート」ボタン（またはファイル選択UI）を配置し、`<input type="file" accept=".json" />` を使用してJSONファイルを読み込ませる。
    *   `FileReader` APIを利用してファイル内容をパースし、キー構成やデータ型がMichiQuestのバックアップデータとして正しい形式か検証する。
    *   妥当であれば、現在のLocalStorage内に履歴や設定等の既存データが存在するか確認する。データが存在する場合は「現在のセーブデータはすべて消去され、インポートしたデータに差し替えられます。この操作は取り消せません。よろしいですか？」といった警告を含む確認ダイアログを表示する。
    *   ユーザーの最終了承を得た上で、現在のLocalStorageの値を上書き保存し、アプリケーションの状態（ReactのState）を更新するためにページをリロードするか初期化処理を再実行する。

## 6. 画面レイアウト・ルーティング
*   React Router等のライブラリは使用せず、`App.jsx` の `currentTab` Stateによってコンポーネント (`ChatThread`, `MapInteractive`, `Settings`, `Notebook` 等) を切り替えるSPA（Single Page Application）として構築。
    *   スマホ利用（バイク搭載のスマホマウント状態など）を想定し、画面の大部分をチャットまたは地図UIに割き、ヘッダー・タブナビゲーションのみを常に表示する。
    *   **サイドバーUI仕様**:
        *   調査記録リストのアクション（再生・削除等）アイコンは、スマホ等（モバイル幅）ではタップしやすさを考慮し常時表示とし、PC等（MD幅以上）ではマウスホバー時のみ表示するレスポンシブデザインとする。
    *   **チャットUI仕様**:
        *   AIアバターの下部に「ミチ・ノマ」というキャラネームラベル（バッジ）を表示し、誰が発言しているかを明確にする。
        *   ユーザーおよびAIの双方のチャット吹き出しの下部に、実際のメッセージ送信時刻（タイムスタンプ）を `HH:mm` 形式で表示し、リアルタイムでの会話感を演出する。
        *   AIアバター画像は、表情視認性向上のため基本サイズを `w-28 h-28 (112px)`、MDサイズ（PC）で `w-32 h-32 (128px)` と大きく設計する。
        *   APIタイムアウト時や通信に失敗した場合、必ずローディング表示を取り下げて適切なエラーテキストを表示し、フリーズを防止する。
