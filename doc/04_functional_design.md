# 04. 機能設計 (Functional Design)

## 1. チャット対話・ミッション生成フロー
1.  **セッション初期化と挨拶**: サイドバー等から新規セッションを作成した際、チャット履歴の初期状態として、定義されたオペレータ（ミチ・ノマ）のランダムな定型文挨拶が1件挿入された状態で開始する。
2.  **自動ミッション生成**: 初期挨拶の表示後、ユーザーの入力を待たずにシステムが自動でGemini APIを呼び出し、ミッションを生成してユーザーに提示する。
    1.  **プロンプト構築 (`api.js`)**:
        *   `App.jsx` から渡された `basePrompt` (基本性格), `prompt1` (ミッション生成用), `prompt2` (画像判定用) などをコンテキストとして組み立て。
        *   システムプロンプト内に `[AREA: {"lat":35.xxx, "lng":139.xxx, "r":100, "name":"地名"}]` 形式での出力を要求。
    2.  **API通信**: Gemini APIに対してリクエストを実行。
    3.  **レスポンスパース (`App.jsx` / `ChatThread.jsx`)**:
        *   応答から `[AREA: ...]` を正規表現で取り出し、地図コンポーネント用の `currentMissionArea` ステートに保存。該当文字列はチャット画面では置換（非表示）とする。また、ミッション履歴のタイトルをこの目的地の名称（例: `📍東京タワー`）に更新する。
        *   同様に `[TITLE: ...]` (称号), `[SCORE: ...]` (得点) をパースし、コレクションとしてLocalStorageに保存。
        *   **クリア判定とリザルト演出**: **60点以上**のスコアを獲得した際に、当該セッションの状態（`isCleared`）を true に更新し永続化する。更新直後には全画面アニメーションを自動発火させず、チャット画面上に「リザルトを見る」手動トリガーボタンを表示する。ユーザーがスコアやコメントを確認した後に当該ボタンをタップすることで初めて「MISSION CLEARED!」等のアニメーションをオーバーレイ表示し、サイドバーの一覧にも「CLEAR!」バッジを表示する。
    4.  **オペレータ対応（通常会話・ミッション再生成）**: ミッション進行中のユーザー発言に対しては、Gemini APIを使用してキャラクターとしての応答（雑談、ヒント提示など）を生成しチャットに表示する。
        *   ユーザーからミッション変更の要求等があり、システムの判断で新しいミッションに切り替える場合、AI応答に `[AREA: {"lat":..., "lng":..., "name":"...", "r":...}]` を含める。
        *   フロントエンド (`ChatThread.jsx`) は応答内にAREAタグを検知した場合、Nominatim API での実在判定を通した上で、地図の目的地 (`currentMissionArea`) と `currentMission` ステートを新しいものへ更新する。AREAタグを含まない場合は状態を更新せず会話のみ進行する。

## 2. ハルシネーション（目的地架空生成）防止フロー
*   **概要**: ミッション生成時に `[AREA]` タグが含まれていた場合、事前にOpenStreetMapなどで実在判定を行う（フェーズ3設計）。
*   **フェーズ3導入前のフォールバック制御**: AIが架空の目的地（緯度経度が提供されないデータなど）を出力した場合や、アップロードした写真のExif情報が不正だった場合、地図コンポーネントにおける描画エラー（クラッシュ）を防ぐため以下の対策を行う。
    *   `lat` / `lng` が存在しない（数値でない）場合は対象エリアの円形（Circle）マーカー等の地図上表示をスキップする。
    *   地図全体を移動させる中心座標（`center`）に対しても、渡された座標配列が不正な場合（`[NaN, NaN]`など）は強制的に初期位置（東京駅周辺など）へフォールバックさせ、アプリの異常終了を回避する。
    *   現在地や写真によるピン（`Marker`）を描画する際にも、座標配列が有効な数値かを検証し、無効な場合はマーカー単体の描画をスキップして地図全体のクラッシュを防ぐ。
*   **ロジック**:
    1.  AIが「富士山頂上のカフェ」などを生成し、`[AREA: {"name": "富士山頂上のカフェ"}]` を出力。
    2.  フロントエンド側で抽出された `name` を使って Nominatim API を叩く。
    3.  結果が空 (API return `[]`) である場合、フロントエンドはエラーを検知し、裏側で「その場所は存在しません、別の実在する場所でミッションを作り直してください」とGeminiに再リクエストを投げる (Auto Retry機構)。
    4.  検証を合格した場所のみがユーザーのチャット画面に表示される。

## 3. 画像報告・アップロード機能
*   **ミッション時の入力UI仕様**: ユーザーが写真報告用のカメラボタンを押した際、「カメラで撮影する」と「写真ライブラリから選ぶ」という2つの選択肢（メニュー）をフロントエンド上で提示する。
    *   **カメラ撮影**: `<input type="file" accept="image/*" capture="environment">` を動的に生成して起動し、カメラアプリを直接立ち上げる。
    *   **写真選択**: `<input type="file" accept="image/*">` (`capture` 属性なし) を動的に生成して起動し、OS標準のファイル/ライブラリ選択ダイアログを表示する。
*   **画像処理フロー**: 選択されたファイルからExif情報（GPS座標）を抽出し、ミッションの判定に利用する。APIへ送信する前には長辺1024px等へクライアントサイドでリサイズを行う。
*   **設定画面（アバター）の手動クロップUI**:
    *   `Settings.jsx` にて画像を選択時、または「画像を調整」ボタン押下時、全画面モーダルコンポーネント（`ImageCropper.jsx` / `react-easy-crop`使用）を表示する。
    *   ユーザーによるドラッグ・ズーム操作で円形クロップの枠を合わせ、Canvas API を用いて指定領域のみを正方形（256x256 等）で切り出し、Base64 で保存する。既存のデフォルト画像や設定済み画像に対しても再調整が可能。

## 4. リプレイモード（フェーズ4）
*   **UI構成**:
    *   サイドバーのセッション・アイテムから「リプレイ開始」を実行。
    *   `App.jsx` で `isReplayMode = true` となり、`ChatThread.jsx` にPropとして渡る。
    *   チャット下部の「送信入力欄」の領域が、動的に「リプレイ・コントローラー（▶️やシークバーのようなUI）」に置換される。
*   **制御ロジック**:
    *   `ChatThread.jsx` 内に `replayIndex` ステートを持つ。
    *   表示する履歴を `chatHistory.slice(0, replayIndex + 1)` として描画。
    *   右矢印キーによるインクリメント、オートプレイ（`setInterval`）による動的インクリメントに対応。
    *   **クリア演出の再現**: `isCleared` が true のセッションをリプレイした場合、最終インデックスまで再生が到達した時点で、手動プレイ時と同様のクリアアニメーションを再表示する。

## 5. 画面レイアウト・ルーティング
*   React Router等のライブラリは使用せず、`App.jsx` の `currentTab` Stateによってコンポーネント (`ChatThread`, `MapInteractive`, `Settings`) を切り替えるSPA（Single Page Application）として構築。
    *   スマホ利用（バイク搭載のスマホマウント状態など）を想定し、画面の大部分をチャットまたは地図UIに割き、ヘッダー・タブナビゲーションのみを常に表示する。
    *   **サイドバーUI仕様**:
        *   ミッション履歴リストのアクション（再生・削除等）アイコンは、スマホ等（モバイル幅）ではタップしやすさを考慮し常時表示とし、PC等（MD幅以上）ではマウスホバー時のみ表示するレスポンシブデザインとする。
    *   **チャットUI仕様**:
    *   AIアバター画像は、表情視認性向上のため基本サイズを `w-28 h-28 (112px)`、MDサイズ（PC）で `w-32 h-32 (128px)` と大きく設計する。
    *   APIタイムアウト時や通信に失敗した場合、必ずローディング表示を取り下げて適切なエラーテキストを表示し、フリーズを防止する。
